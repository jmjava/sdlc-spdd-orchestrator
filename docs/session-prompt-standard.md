# Session Prompt Standard

This is the unified prompt contract for SDLC-SPDD sessions. It bridges all three layers:

- **Planning** — narrative layer (`ROADMAP.md`, `milestone-*.md`, `requirements/`, `requirements/milestones/`, `session-notes/`)
- **SPDD** — governed design contract (`spdd/canvas/<WORK-ID>.md`)
- **SDLC** — lifecycle discipline (phase commands, session briefs, memory)

Use this page when you need copy-paste prompts that stay consistent across team members, Cursor, and Copilot.

**Default here.** For day-to-day agent work, start with this page. Use the concept-specific standards below only when you need deeper prompts for that layer.

| Need | Open |
|------|------|
| Copy-paste prompts | **This page** |
| Step order (1–13) | [Workflow](workflow.md) |
| Rules, scripts, phase checklists | [Daily runbook](daily-runbook.md) |
| Planning → SPDD → SDLC loop | [Three-part operating path](three-part-operating-path.md) |

**Resume prompt source of truth:** the **Resume Prompt** section in `agent-context/sessions/current-session.md`, generated by `start-agent-session.sh`. Do not paraphrase it in other docs or chat — paste it verbatim.

For the full Planning → SPDD → SDLC operating loop (not just prompts), see [Three-part operating path](three-part-operating-path.md).

## Which Prompt Standard?

Three prompt standards exist. They overlap by design at layer boundaries. Use this guide to pick the right one.

### Quick rule

| Your question | Open this |
|---------------|-----------|
| Starting, continuing, or ending an agent session? | **This page** (Session) |
| Governing *what* to build via the canvas? | [SPDD prompt standard](spdd-prompt-standard.md) |
| Tracking *why* and *progress* via roadmap/milestones/notes? | [Planning prompt standard](planning-prompt-standard.md) |

### Decision tree

    What are you doing right now?
    |
    +-- Starting or resuming work in a new agent chat
    |     -> Session (resume prompt from start-agent-session.sh)
    |
    +-- Working through a Work ID (plan, code, review, handoff)
    |     -> Session (default for the full session loop)
    |     -> SPDD (when the question is canvas-specific — see below)
    |     -> Planning (when the question is narrative/progress — see below)
    |
    +-- Canvas contract question only (no full session context needed)
    |     -> SPDD
    |
    +-- Roadmap, milestone, or session-note question only
          -> Planning

### When to drill into SPDD

Open [SPDD prompt standard](spdd-prompt-standard.md) when the prompt must reference the **REASONS Canvas contract**:

- Validate canvas structure or readiness for coding
- Compare Requirements to Jira/GitHub acceptance criteria (alignment)
- Harden design with `/sdlc-spdd-architect`
- Check whether a diff stays inside one approved Operation
- Decide prompt-update vs sync after a change
- Run the one-operation code → review → sync loop

### When to drill into Planning

Open [Planning prompt standard](planning-prompt-standard.md) when the prompt must reference **delivery narrative artifacts**:

- Map milestone checklist items to Work IDs (`create-work-from-milestone.sh`)
- Plan with `@ROADMAP.md` and `@milestone-*.md` context
- Capture to `session-notes/`, milestone files, or `ROADMAP.md`
- Refresh the managed roadmap summary (`sync-roadmap-from-spdd.sh`)
- Import historical session notes into memory

### Boundary moments (use more than one)

Some steps touch two layers. Start with Session; add the focused standard when you need layer-specific templates.

| Moment | Start here | Also open |
|--------|------------|-----------|
| Start session on milestone work | Session (resume prompt) | Planning for `@ROADMAP.md` / `@milestone-*.md` refs |
| End session, capture handoff | Session (capture template) | Planning for `--milestone` / `--roadmap-note`; SPDD for `--next` review/sync command |
| Milestone item → new Work ID | Planning (`create-work-from-milestone.sh`) | SPDD for plan/architect prompts the script prints |
| Acceptance criteria changed | SPDD (`/sdlc-spdd-prompt-update`) | Planning to update milestone or roadmap narrative |
| Roadmap summary out of date | Planning (`sync-roadmap-from-spdd.sh`) | SPDD if canvas Metadata is wrong |

### Why overlap exists

Session prompts **compose** planning and SPDD references into one handoff (for example the resume prompt includes `@ROADMAP.md`, `@milestone-*.md`, and `@spdd/canvas/<WORK-ID>.md`). SPDD and Planning standards **decompose** those same concerns when you need the full template set for one layer only.

If you are unsure: **open Session first**, then follow the link to SPDD or Planning when your question narrows to canvas governance or delivery narrative.

## Required Prompt Fields

Every session prompt should include as many of these as apply:

| Field | Required | Example |
|-------|----------|---------|
| Work ID | Yes, when work exists | `FEAT-001-order-status-api` |
| Session brief | Yes, at session start | `@agent-context/sessions/current-session.md` |
| Canvas | Yes, when Work ID exists | `@spdd/canvas/<WORK-ID>.md` |
| Phase | Yes | `code`, `review`, `plan`, etc. |
| Planning context | When milestone work | `@ROADMAP.md`, `@milestone-1.md` |
| Session notes | When continuing same day | `@session-notes/YYYY-MM-DD.md` |
| Next command | At handoff | `/sdlc-spdd-review @spdd/canvas/<WORK-ID>.md` |

## Session Lifecycle

    resync (optional) -> start-agent-session -> resume prompt -> during-session prompts -> capture-session-memory

Scripts:

    ./scripts/sdlc-spdd/resync-agent-session.sh --target . --work-id <WORK-ID> --check-only
    ./scripts/sdlc-spdd/start-agent-session.sh --target . --work-id <WORK-ID> --phase <phase> [--milestone milestone-1.md]
    ./scripts/sdlc-spdd/capture-session-memory.sh --target . --work-id <WORK-ID> --phase <phase> --summary "..." --validation "..." --next "..."

## Start of Session

### Step 1 — Generate the session brief

    ./scripts/sdlc-spdd/start-agent-session.sh --target . --work-id <WORK-ID> --phase <phase>

With an explicit milestone:

    ./scripts/sdlc-spdd/start-agent-session.sh --target . --work-id <WORK-ID> --phase code --milestone milestone-1.md

The script writes `agent-context/sessions/current-session.md` and includes a **Resume Prompt** section. Prefer that generated prompt — it reflects artifact status and planning files present on disk.

### Step 2 — Paste the resume prompt

Standard template (also generated by `start-agent-session.sh` when artifacts exist):

    For <WORK-ID>, read @agent-context/sessions/current-session.md first.

    For <WORK-ID>, read @spdd/canvas/<WORK-ID>.md, @agent-context/features/<WORK-ID>/progress-log.md, and @agent-context/memory/known-pitfalls.md.

    Also read @ROADMAP.md, @milestone-1.md, and @session-notes/YYYY-MM-DD.md when they exist.

    Continue in the <phase> phase using the hybrid SDLC Agents + SPDD workflow.
    Recommended command: <recommended command from session brief>

### Step 3 — Morning status check (optional)

Use when you want a status summary before acting:

    For <WORK-ID>, read @agent-context/sessions/current-session.md first.

    For <WORK-ID>, read @spdd/canvas/<WORK-ID>.md, @agent-context/features/<WORK-ID>/progress-log.md, and @agent-context/memory/known-pitfalls.md. Summarize:
    - current status
    - next approved operation
    - open risks
    - recommended SDLC-SPDD command

    Also read @ROADMAP.md, the active @milestone-*.md file, and today's @session-notes/YYYY-MM-DD.md if they exist.

## During Session

### Context-preserving question

    For <WORK-ID>, read @spdd/canvas/<WORK-ID>.md and @agent-context/sessions/current-session.md before answering. <question>

### Continue interrupted work

    For <WORK-ID>, read the current session brief, canvas, progress log, review report, sync log, and current diff. Tell me the next SDLC-SPDD command.

### Check scope before coding

    For <WORK-ID> operation T01, inspect the canvas and current diff. Are any changes outside the approved operation?

### Milestone-aware planning

    /sdlc-spdd-plan @requirements/<topic>.md @ROADMAP.md @milestone-1.md

Or:

    /sdlc-spdd-plan Jira ABC-123 for milestone-1.md. Link this Work ID to the milestone and update the canvas Metadata.

### Behavior change (SPDD rule)

    /sdlc-spdd-prompt-update @spdd/canvas/<WORK-ID>.md

Do not edit application behavior until the canvas is updated.

### Accepted implementation drift

    /sdlc-spdd-review @spdd/canvas/<WORK-ID>.md
    /sdlc-spdd-sync @spdd/canvas/<WORK-ID>.md

## End of Session

### Capture handoff (prompt)

    For <WORK-ID>, summarize completed work, tests, open risks, decisions, pitfalls, reusable patterns, and the next command.

### Capture handoff (script)

    ./scripts/sdlc-spdd/capture-session-memory.sh \
      --target . \
      --work-id <WORK-ID> \
      --phase <phase> \
      --summary "<completed work>" \
      --validation "<tests or checks>" \
      --decisions "<decisions, if any>" \
      --pitfalls "<pitfalls, if any>" \
      --patterns "<patterns, if any>" \
      --milestone milestone-1.md \
      --roadmap-note "<roadmap-level progress>" \
      --next "<next command>"

This updates:

- `agent-context/memory/session-history.md`
- `agent-context/features/<WORK-ID>/progress-log.md`
- `session-notes/YYYY-MM-DD.md`
- `milestone-*.md` when `--milestone` is provided or auto-detected from `milestone-*.md`
- `ROADMAP.md` when `--roadmap-note` is provided

## Triage (no Work ID yet)

    Triage this request before planning. Identify whether it is FEAT, BUG, REF, SPIKE, DOC, TEST, or CHORE; propose a Work ID; list missing information; and tell me whether we can safely run /sdlc-spdd-plan.

If ready:

    /sdlc-spdd-plan <requirement, Jira issue, GitHub issue, or @requirements/file.md>

## Anti-Patterns

Avoid these — they lose context across all three layers:

| Bad prompt | Why it fails |
|------------|--------------|
| `Continue.` | No Work ID, no artifacts, no phase |
| `What now?` | No session brief or canvas reference |
| `Fix it.` | No operation boundary or SPDD contract |
| Coding before architect | Skips SDLC architecture-first gate |
| Behavior change without prompt-update | Violates SPDD prompt-first rule |
| Ending session without capture | Loses SDLC durable memory and session notes |

## During-Session Rules (always apply)

1. Keep the active Work ID in every prompt.
2. Load only phase-relevant context.
3. Use the REASONS Canvas as the SPDD contract.
4. Implement one approved Operation at a time.
5. Use `/sdlc-spdd-prompt-update` before behavior changes.
6. Use `/sdlc-spdd-sync` after accepted implementation drift.
7. Include milestone and roadmap context when the work belongs to a milestone.
